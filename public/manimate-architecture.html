Architecture decision

Small core, explicit heavy paths

Manimate should stay simple: Claude owns the workspace, the backend observes canonical files, SQLite stores UI truth, and the UI reads small APIs. Heavy operations like library full-text search are kept, but they must be explicit and low-frequency.

1. The Principle

Keep the product architecture. Reduce the infra architecture. The system should not invent a second synchronization framework around Claude. It should observe Claude's outputs and make those outputs readable.

Claude owns work

Claude runs in the project directory and freely edits/renders files.

execution

Backend owns observation

The backend watches known outputs and writes normalized state to SQLite.

observer

SQLite owns UI truth

The UI does not infer state by scanning files or parsing tool events.

state

2. Complexity To Delete Or Avoid

Component Decision Reason
session_index table Delete / do not add Make sessions small enough to serve as the index.
artifact_snapshots Delete / do not add Artifact metadata is enough. Read specific files on demand.
Separate cloud_sync table Defer Simple remote fields are enough until multi-device sync becomes central.
BroadcastChannel Delete / do not add Hidden tabs can sleep and refresh on focus.
/api/events/stream Defer The active chat stream already exists. Past sessions can fetch SQLite state.
General job queue framework Delete / do not add A simple runs table plus local process tracking is enough.
Lazy thumbnail generation on read Delete GET routes should serve files, not run ffmpeg.
Library full-text search Keep Plan/script search is useful for accurate review and reuse.

4. SQLite Schema

The key simplification is that sessions is small and can be listed directly. Large text and media do not belong in the session list response.

sessions

  • id primary key
  • session_number
  • title
  • status
  • model
  • aspect_ratio
  • voice_id
  • video_path
  • last_video_url
  • remote_url
  • remote_status
  • last_user_activity_at
  • created_at, updated_at

messages

  • id primary key
  • session_id indexed
  • role
  • content
  • metadata_json
  • created_at

  • id primary key
  • session_id indexed
  • status
  • pid
  • pgid
  • started_at
  • heartbeat_at
  • finished_at

  • id autoincrement
  • session_id indexed
  • run_id indexed
  • type
  • message
  • payload_json
  • created_at

artifacts

  • id primary key
  • session_id indexed
  • kind
  • path
  • size
  • mtime
  • version
  • created_at

settings

  • key primary key
  • value_json
  • updated_at
Current code still stores plan/script/subtitles/chapters on sessions. That can remain temporarily, but normal list APIs must not return those fields.

5. API Shape

Endpoint Audience Returns Rule
GET /api/sessions Sidebar, feedback picker Small session list fields only No plan/script. No filesystem stat loop.
GET /api/sessions?include_search_content=1 Library search Sessions plus plan/script content Allowed to be heavy. Must be explicit and low-frequency.
GET /api/sessions/:id/messages Session page Transcript, current session artifacts, active run One session only.
POST /api/chat Web UI SSE stream for active generation Creates run, streams progress, persists events.
POST /api/tool/generate CLI / agents SSE-compatible generation pipeline Same backend path as UI.
GET /api/thumbnail Library card Existing thumbnail file or 404 Never runs ffmpeg on read.
GET /api/files Preview/download Specific file stream Requires session-scoped path resolution.

7. Runs, Not A General Queue

Manimate does not need a queue framework. It needs a durable record of the local process and a way to recover after restart. 

run states:
  queued -> running -> completed
                 |-> failed
                 |-> canceled
                 |-> interrupted

startup recovery:
  find runs where status in ('queued', 'running')
  if pid is missing, dead, or too old:
    mark interrupted

cancel:
  kill process group
  mark canceled
  append event

8. Library Full-Text Search

Keep it. Searching plan and script content is useful for accurately finding prior implementations. The rule is that it is an explicit Library path, not the default sidebar path.

Default list

Used by sidebar and feedback picker.

small frequent safe

GET /api/sessions

Library search

Used only by Library video search.

heavy explicit

GET /api/sessions?include_search_content=1

Current measured payloads: default list is about 288 KB for 1006 sessions. Library search is about 27.3 MB because it includes plan and script content. That is acceptable only because the Library refresh is low-frequency.

9. Storage Rules

SQLite

Stores UI state, messages, runs, events, artifact metadata, and simple remote status.

Filesystem

Stores Claude workspace and actual artifacts: scripts, plans, media, thumbnails, inputs.

R2

Stores shareable outputs and remote snapshots. It does not replace local SQLite.

UI

Stores temporary view state only. It starts/cancels runs and displays SQLite-backed state.

10. Current Implementation Changes

Done in code

/api/sessions is lightweight by default.

Sidebar uses default list and polls every 10 seconds only when visible.

Library full-text search is kept but refreshes every 60 seconds while visible.

Thumbnail GET route no longer generates thumbnails.

Cloud sync uses existing thumbnail only.

Add pid/pgid/heartbeat to run recovery if not already fully wired.
